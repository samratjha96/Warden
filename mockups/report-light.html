Browser interaction tooling constructs and evaluates function bodies dynamically. Intended for browser-context automation — but if attacker-controlled input reaches these paths, arbitrary code execution follows with no sandbox boundary.

The extension requests debugger, tabs, activeTab, storage, alarms, and webNavigation. Host permissions are loopback-only, but debugger + tabs together enable near-total control over all open browser tabs — including credential fields and authenticated sessions.

Remediation

If evaluation is required, approve only under strict isolation controls:

1
Run only in an isolated VM or container — no privileged corp credentials, no host mounts.
2
Enforce outbound network allowlists — permit only provider domains strictly required for your use case.
3
Disable the OTel diagnostics plugin unless explicitly needed and audited.
4
Replace curl | bash installers with pinned artifacts and SBOM/checksum verification.
5
Restrict exec tooling by strict allowlist — require explicit approvals before any shell execution.
6
Verify gradle-wrapper.jar in CI via Gradle wrapper validation and checksum policy.
7
Require branch protection and signed-release verification before internal adoption of tagged releases.
