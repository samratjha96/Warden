Code Safety

HIGH
HIGH Dynamic code execution via eval / new Function in browser tooling

Browser interaction tooling constructs and evaluates function bodies dynamically. This is the intended mechanism for browser-context evaluation but is a high-risk primitive — if attacker-controlled input reaches these paths, arbitrary code execution is possible.

References

HIGH Shell/system execution framework — central to node-host functionality

Process execution is a core feature, not an incidental capability. An allowlist/policy mechanism exists but is non-trivial and relies on correct configuration to prevent abuse.

References
src/node-host/invoke.ts:122Spawns OS commands
src/node-host/invoke-system-run.ts:180Policy/allowlist machinery
HIGH macOS installer uses curl … | bash remote script pattern

The macOS app installer builds and executes a command of the form curl ... | bash. This pattern is inherently vulnerable to man-in-the-middle attacks and TOFU (trust on first use) failures — the executed script has full host privileges.

Remediation Suggestions

If evaluation is required, APPROVE WITH CONDITIONS only under strict controls:

1
Run only in an isolated VM or container with no privileged corp credentials or host mounts.
2
Enforce outbound network allowlists — permit only the provider domains strictly required for your use case.
3
Disable or remove optional telemetry exporters (diagnostics-otel plugin) unless explicitly needed.
4
Disable remote installer one-liners — use a pinned artifact with checksum/SBOM verification instead of curl | bash.
5
Restrict exec tooling by policy (deny/strict allowlists) and require explicit approvals before any shell execution.
6
Verify gradle-wrapper.jar provenance in CI using Gradle wrapper validation and checksum policy.
7
Prefer tagged releases and pinned commits. Require branch protection and signed-release verification before internal adoption.
